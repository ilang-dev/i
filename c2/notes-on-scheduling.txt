> there are a lot of complicated aspects to the scheduling that I'd like to
> capture as cleanly as possible. the loop splitting is somewhat simple in that
> it does not depend on any other scheduling. the loop ordering then only
> depends on the splitting (to determine what loops there are), but compute-at

> loops get fused as aggressively as possible (with all the details that go
> with loop fusion), and second, storage gets folded as aggressively as
> possible, so we have to adjust buffer allocations. in the past, I tried to do
> the loop fusion with a constructive approach where I would build the full
> parent loop stack but then build on a "fragment" from the child loop which
> just got injected at the compute-at site. this is semantically incorrect, but
> also messy to implement.  this time around I'd rather literally construct the
> loops and then merge them one at a time where the merge code can be very
> direct and easy to review. what is your suggestion for handling all the
> fallout from the compute-at directives?


My recommendation is to split the problem very hard into two parts:

1. resolve compute_at into explicit placement facts on the stage graph
2. only then construct and merge loop trees

The mistake in the old “inject fragment into parent stack” approach is that
it tried to resolve placement and construct execution at the same time.

I would do this instead.

Phase 1: Resolve Placement Declaratively

Treat each compute_at directive as a request of the form:

- producer P
- consumer C
- site = some loop of C

Normalize all of those into explicit edge-placement facts.

Then reconcile them globally:

- if P has one consumer, easy
- if P has multiple consumers, all requests must agree or one must dominate
  the others in a way your semantics explicitly allow
- otherwise reject the schedule

This phase should answer only:

- which stages are fused into which kernel
- for each fused producer, where it is placed relative to its consumer
- which edges remain materialized across kernel boundaries

This is the phase where compute_at fallout should mostly live. Not in loop
construction.

Once this is done, you should have something like an internal kernel plan:

- kernel membership
- stage placement site inside the kernel
- external reads/writes
- materialized boundaries

At that point, compute_at has done its job.

Phase 2: Build Full Loop Nests Per Stage

Inside one kernel, first build each stage’s full standalone loop nest from
only:

- semantic axes
- splits
- order

No fusion logic yet. No fragments. Just “what would this stage’s loop nest be
by itself?”

That gives you clean, reviewable raw material.

Phase 3: Merge Loop Trees One Placement At A Time

Then do exactly what you said: literally merge loop trees.

For a producer placed at a consumer site:

- descend the consumer loop tree to the target site
- compare the producer’s outer loop prefix against the consumer loops above
  that site
- merge identical loops directly
- when they diverge, either reject or insert producer-only loops locally,
  according to the legality you want
- then splice the producer body in at that exact point

This is much better than “fragment injection” because the merge code can be
very literal:

- loop with same axis/extents/order meaning => unify
- otherwise => not the same loop
- producer-only remainder => nest under the insertion point

That code becomes easy to inspect.

Phase 4: Derive Storage After Placement Is Fixed

Do not try to reason about storage folding while resolving compute_at.

First fix placement.  Then derive storage from placement.

For each stage value, compute:

- is it internal to one fused kernel or crossing a kernel boundary?
- if internal, is it consumed immediately at one placement site or across
  multiple dynamic iterations?
- what loops separate definition from last use?

Then choose:

- no buffer at all if the value is purely inlined into one use site
- a folded buffer whose rank is the number of live varying loops
- a full buffer only when necessary

The right mental model is:

- compute_at determines placement
- placement determines lifetime
- lifetime determines storage
- storage determines folding/allocation shape

That ordering keeps the logic clean.

What I Would Represent Internally

I would introduce a private planning layer with something close to:

struct KernelPlan { stages: Vec<StageId>, placements: Vec<StagePlacement>,
inputs: Vec<ValueId>, outputs: Vec<ValueId>, }

struct StagePlacement { stage: StageId, at: Site, }

struct BufferPlan { value: ValueId, kind: BufferKind, shape_from_loops:
Vec<LoopId>, }

Not as public IR. Just as the bridge between scheduled graph and loop IR.

Practical Rule

Do not let compute_at directly mutate loops.  Let it only produce placement
constraints.

Then:

- resolve constraints
- form kernels
- build standalone loop nests
- merge nests mechanically
- derive storage mechanically from the merged result

That is the cleanest way I know to contain the fallout. It matches your
instinct to make loop merging direct and reviewable, while keeping the
hardest semantic questions out of the merge code.
